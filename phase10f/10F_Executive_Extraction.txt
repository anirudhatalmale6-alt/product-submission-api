SECTION A: TOP 10 CURRENT TRUTHFULNESS FAILURES
================================================================================

FAILURE 1: JOBS ARE INVISIBLE
  What: When you click any action button, a job is created internally but
        you never see it. There is no jobs table, no jobs panel, no way
        to answer "what has the system done today?"
  Where: Manager dashboard, Operations Centre, NOC — none show jobs.
  Owner risk: You have no way to know if an action you triggered actually
        ran, is still running, failed silently, or completed. You are
        operating blind on system activity.
  Classification: SHALLOW
  10F fix: Real-time Recent Jobs table on the manager dashboard and NOC.
        Every action produces a visible, persistent, clickable job entry.

FAILURE 2: APPROVAL QUEUE IS THEATER
  What: The approval queue exists as UI and API but contains only test
        data from setup scripts. No agent has ever created a real approval
        request. No workflow gate is connected to approval decisions.
  Where: Manager dashboard governance section, /api/approvals endpoint.
  Owner risk: You see an "approvals" badge that implies governance is active,
        but nothing in the system actually asks for your permission before
        acting. The traffic light system (GREEN/AMBER/RED) classifies actions
        but does not enforce approval gates.
  Classification: MISLEADING
  10F fix: Wire AMBER and RED actions to create real approval requests.
        The action physically cannot execute until you approve or reject it.
        Standing approvals can auto-approve specific AMBER actions you trust.

FAILURE 3: EVENTS VANISH ON RESTART
  What: The primary event feed is an in-memory Python list. When the
        manager agent restarts, all events are lost. Redis Streams and
        PostgreSQL event_history exist but contain near-zero real data
        (1-3 test entries each).
  Where: Event feed on manager dashboard, /api/events endpoint.
  Owner risk: If the system restarts (update, crash, maintenance), you
        lose all record of what happened. There is no persistent operational
        story. You cannot audit what occurred overnight or over a weekend.
  Classification: MISLEADING
  10F fix: All events written to PostgreSQL on creation. Survive any
        restart. Queryable event timeline showing the last 24 hours,
        7 days, or 30 days of operational activity.

FAILURE 4: AGENT HEARTBEATS ARE 6 DAYS STALE
  What: The PostgreSQL agents table shows agents as "ACTIVE" with
        heartbeat timestamps from 6 days ago. There is no timeout
        detection. An agent that crashed 6 days ago still shows as active.
  Where: Agent health grid on manager dashboard, /api/agents endpoint,
        PostgreSQL agents table.
  Owner risk: You look at the agent grid and see green dots, but you
        have no idea if those agents are actually running right now. The
        status display is frozen in time and does not reflect reality.
  Classification: SHALLOW
  10F fix: Heartbeat timeout detection every 60 seconds. If no heartbeat
        within 5 minutes, agent status auto-transitions to UNREACHABLE.
        All transitions logged. Agent cards show "last seen X ago" with
        color-coded freshness.

FAILURE 5: ASYNC ACTIONS SHOW FALSE COMPLETION
  What: When you click "SEO Audit", the system reports "completed" in
        65 milliseconds with the message "Audit started". The actual SEO
        audit has not started yet — what completed is the HTTP request to
        the SEO agent, not the audit itself.
  Where: Every dispatch button on the manager dashboard.
  Owner risk: You believe the audit is done when it has not even begun.
        If the actual audit fails 30 seconds later, you will never know.
        The dashboard gives you false confidence in completed work.
  Classification: SHALLOW (approaching MISLEADING for async operations)
  10F fix: Jobs stay in "running" state until the real work completes.
        Progress updates show actual progress. If no completion callback
        arrives within the timeout window, the job transitions to
        "completed_unverified" — a visually distinct state that tells you
        the dispatch was acknowledged but the outcome is unconfirmed.

FAILURE 6: ACTION RECEIPTS ARE OVERWRITTEN
  What: When you click a button, a small inline result appears below
        the button. If you click another button, the previous result
        disappears. There is no history of actions you have taken.
  Where: Quick actions grid on manager dashboard.
  Owner risk: You cannot answer "what did I do 10 minutes ago?" or
        "which actions have I triggered today?" Every action receipt is
        temporary and lost when the next action runs.
  Classification: SHALLOW
  10F fix: Persistent action receipts in a dedicated Recent Actions panel.
        Every action creates a receipt that stays visible. Last 50 receipts
        always available. Click any receipt to see the linked job.

FAILURE 7: PIPELINE HISTORY IS LOST
  What: The 6-stage pipeline (Research->Content->SEO->Affiliate->Social->
        Engagement) runs and produces detailed step results, but this data
        is not saved anywhere persistent. Previous pipeline runs cannot
        be reviewed.
  Where: Pipeline section on manager dashboard.
  Owner risk: You run a pipeline, it finishes, and the results vanish.
        You cannot compare runs, identify patterns, or review what a
        previous pipeline actually did. There is no production history.
  Classification: SHALLOW
  10F fix: Every pipeline run and its step details saved to PostgreSQL.
        Pipeline history panel shows past runs with step-by-step state
        breakdown. Click any run to see full detail including which
        steps succeeded, failed, or were skipped (with reasons).

FAILURE 8: THREE PARALLEL DATA STORES FOR EVERYTHING
  What: The same types of data (jobs, events, approvals) are stored in
        up to three different places simultaneously with no synchronization.
        Jobs: JSON file + PostgreSQL tasks table.
        Events: in-memory list + Redis Streams + PostgreSQL event_history.
        Approvals: JSON file + PostgreSQL approvals table.
  Where: Throughout the entire backend.
  Owner risk: This is the root cause of most other failures. When data
        lives in multiple places, no single surface can be trusted to
        show the complete picture. Different dashboards may show different
        versions of reality depending on which store they query.
  Classification: STRUCTURALLY FALSE (the architecture itself prevents truth)
  10F fix: Single source of truth: PostgreSQL. One unified ops layer
        (truthful_ops.py) that all code writes through. Redis for pub/sub
        notifications only. JSON files deprecated to backup cache.

FAILURE 9: NO WORKFLOW STATE TRACKING PER CONTENT ITEM
  What: There is no system that tracks what happened to a specific article
        after it was created. Did SEO audit it? Did Affiliate scan it?
        Was it promoted on social? Each agent works in isolation with no
        shared lifecycle state.
  Where: Nowhere — this capability does not exist.
  Owner risk: You cannot answer "what stage is this article at?" or
        "which articles are waiting for SEO review?" Content moves through
        the system invisibly. You have no pipeline visibility per item.
  Classification: MISSING (not shallow or misleading — it does not exist)
  10F fix: ops_workflow_items table tracks each content item through 6
        stages (research, content, SEO, affiliate, social, engagement).
        Each stage has a canonical state. Visual pipeline shows where
        each item sits and what is blocking progress.

FAILURE 10: ALERTS HAVE NO LIFECYCLE
  What: When something goes wrong (agent unhealthy, social instability,
        job failure), an event is logged but there is no structured alert
        system. Warnings can repeat indefinitely without resolution.
        Issues can quietly disappear without being resolved.
  Where: Safety alerts section, event feed, agent health grid.
  Owner risk: Ghost warnings persist without explanation. Real problems
        can vanish from view without being fixed. There is no way to
        distinguish "active problem requiring your attention" from
        "already resolved" from "acknowledged but deferred."
  Classification: SHALLOW
  10F fix: Stateful alert lifecycle: active -> resolved / stale / suppressed.
        Every alert has severity, affected component, whether user action
        is required, and whether auto-retry is in progress. Resolution
        is logged with reason. Stale alerts auto-flag after configurable
        timeout. No ghost warnings.

SECTION C: TOP 5 OWNER-FACING IMPROVEMENTS
================================================================================

1. WHEN I CLICK A BUTTON, I WILL SEE THE FULL STORY

   Right now: You click "SEO Audit", see a tiny result that says "Audit
   started" for 65 milliseconds, and then it vanishes when you click
   something else.

   After 10F: You click "SEO Audit" and a job appears in the Recent Jobs
   table. It shows "queued", then "running" with a progress bar, then
   "completed" with a result summary — or "failed" with an error
   explanation. The job stays visible. You can click it to see the full
   timeline of what happened. An action receipt appears in the Recent
   Actions panel confirming what you did and when.

2. WHEN I LOG IN AFTER BEING AWAY, I CAN SEE WHAT HAPPENED

   Right now: If you log in after a weekend, the dashboard looks the same
   as when you left. There is no record of what the system did while you
   were gone (events were in-memory and lost on restart).

   After 10F: The Event Timeline shows every significant thing that happened
   since your last visit — jobs that ran, approvals that were auto-approved,
   alerts that were raised, agents that went unreachable and recovered,
   pipeline runs that completed or failed. All in chronological order with
   plain English descriptions.

3. WHEN SOMETHING NEEDS MY APPROVAL, I WILL SEE EXACTLY WHAT AND WHY

   Right now: The approval queue contains only test data. No agent actually
   asks for your permission before acting.

   After 10F: When the system wants to publish a draft post, run a live
   social campaign, or apply an SEO auto-fix, an approval card appears
   with: what it wants to do, what will change, what the current state is,
   what the new state will be, why it recommends this, what risk level it
   carries, and what happens if you approve or reject. You make an informed
   decision with full context.

4. WHEN AN AGENT IS DOWN OR DEGRADED, I WILL KNOW IMMEDIATELY

   Right now: Agent cards can show green "ACTIVE" status even when the
   agent has not responded in 6 days. There is no timeout detection. You
   could be looking at a healthy-seeming dashboard while half the agents
   are actually crashed.

   After 10F: Agent cards show live status with "last seen X ago." If an
   agent misses its heartbeat for 5 minutes, it turns red and says
   "UNREACHABLE — last seen 7 minutes ago." If it is running in fallback
   mode (using templates instead of AI), a yellow badge says "FALLBACK."
   An alert is raised that you can see in the alerts panel. When the agent
   recovers, the transition is logged.

5. WHEN A WORKFLOW STEP IS SKIPPED OR FAILS, I WILL KNOW WHY

   Right now: If a pipeline step fails or gets skipped, there may be no
   visible trace. The pipeline just stops or moves on silently.

   After 10F: Every pipeline step shows its state in the visual pipeline
   display. If a step is skipped, it explicitly says why ("no trends
   found", "social token expired", "operator paused social posting").
   If a step fails, it shows the error and whether it is retryable.
   The step state is logged in the timeline so you can see exactly what
   happened at each stage.

SECTION D: TOP 5 HIGHEST-RISK IMPLEMENTATION AREAS
================================================================================

RISK 1: JOB CENTRE MIGRATION (JSON -> PostgreSQL)
  What: Moving the active job tracking system from a JSON file to PostgreSQL.
  Why risky: The JSON-based job centre is the one component that currently
        works — tracked_dispatch creates real jobs with real IDs. Breaking
        it during migration would make the dashboard worse, not better.
  Risk reduction:
        - Dual-write period: write to both JSON and PostgreSQL simultaneously
        - Verify PostgreSQL data matches JSON after each operation
        - Old /api/jobs/* endpoints continue to work during migration
        - Only switch dashboard to PostgreSQL source after verification period
  Rollback: Revert job_centre.py to previous version. JSON file is still
        intact and becomes primary again immediately. Zero data loss.

RISK 2: APPROVAL QUEUE BLOCKING REAL ACTIONS
  What: Wiring AMBER/RED actions to actually block execution pending approval.
  Why risky: If the approval gate has a bug, it could block ALL actions
        including ones that should be GREEN (auto-allowed). An operator
        who is not actively monitoring could have work silently stall.
  Risk reduction:
        - GREEN actions are never gated (hardcoded bypass)
        - AMBER actions that match standing approvals auto-approve instantly
        - Approval expiry: if no decision within 24 hours, alert raised
        - Manual override endpoint: POST /api/ops/approvals/force-approve
        - Approval gating deployed per-action-type incrementally, not all at once
  Rollback: Remove approval check from tracked_dispatch. All actions
        revert to auto-execute (the current behavior). No blocked work.

RISK 3: EVENT PERSISTENCE PERFORMANCE
  What: Writing every operational event to PostgreSQL in real-time.
  Why risky: If event volume is high during a monitoring cycle (many
        WordPress changes, many agent actions), the database write load
        could slow down the manager agent's response time.
  Risk reduction:
        - Async write queue with batch flushing (write 50 events at once)
        - In-memory cache still serves reads (PostgreSQL is backup truth)
        - Event table has 90-day retention with auto-cleanup
        - If PostgreSQL write fails, event still exists in memory + logged to disk
  Rollback: Remove PostgreSQL write call from event_engine._log_event().
        Events revert to in-memory only (current behavior).

RISK 4: HEARTBEAT TIMEOUT FALSE POSITIVES
  What: Auto-transitioning agents to UNREACHABLE after 5 minutes of no heartbeat.
  Why risky: If an agent is busy with a long-running task (e.g., bulk SEO audit
        taking 10 minutes), it might not send heartbeats during execution.
        The system would incorrectly mark it as UNREACHABLE.
  Risk reduction:
        - Timeout check uses both heartbeat AND HTTP health check
        - If heartbeat is stale but HTTP /api/status responds, agent is "busy" not "unreachable"
        - Timeout threshold is configurable (start at 5 min, adjust if needed)
        - UNREACHABLE status triggers an alert but does NOT stop or restart the agent
  Rollback: Disable the background heartbeat checker. Agent cards revert
        to showing the raw heartbeat data without timeout interpretation.

RISK 5: NOC DASHBOARD REBUILD BREAKING EXISTING ACCESS
  What: Rebuilding the NOC dashboard with 6 new panels.
  Why risky: The NOC is currently a working dashboard showing real
        infrastructure health (Redis, PostgreSQL, event bus, kill switch).
        A bad rebuild could lose those working health checks.
  Risk reduction:
        - Keep existing infrastructure health panels intact
        - Add new panels below/alongside existing ones
        - Build the new NOC as a separate file first (ops_noc_v2.html)
        - Test in parallel before replacing the current version
        - Screenshot comparison before and after
  Rollback: Swap back to the original operations_noc_dashboard.html.
        The file is version-controlled and backed up.

SECTION E: PLAIN-ENGLISH DEFINITION OF "10F COMPLETE"
================================================================================

10F is complete when the dashboard tells the truth.

Specifically:

When you click a button, you see what happened — not a fleeting message
that vanishes.

When an agent is down, the dashboard says it is down — not green and
healthy.

When the system wants to do something risky, it asks you first — with
enough information for you to say yes or no with confidence.

When you log in after being away, you can see what happened while you
were gone — jobs that ran, things that failed, alerts that were raised,
decisions that were made.

When a pipeline runs, you can see every step — what succeeded, what
failed, what was skipped, and why.

When something is a recommendation, it looks different from something
the system has actually done. When data is estimated or forecast, it
is labelled as such — not presented as if it were measured fact.

When a warning appears, it has a clear status — active, resolved, or
stale. It does not repeat forever without explanation, and it does not
quietly vanish without being resolved.

The one-sentence version:

10F is complete when you can trust what the dashboard shows you, act on
it with confidence, and verify after the fact that the system did what
it said it did.

------------------------------------------------------------------------
SURFACE 1: MANAGER DASHBOARD ADDITIONS
------------------------------------------------------------------------

Where it appears:
  Below the existing quick actions grid on the main manager dashboard
  (manager_dashboard.html, served at /agents/manager/).

What you will see:
  Two new panels:
  a) Recent Jobs table — last 50 jobs in a scrollable table.
     Columns: Job ID (clickable), Agent, Action, Status (color badge),
     Duration, Created time, Summary.
  b) Recent Actions list — last 50 action receipts.
     Each shows: action name, time, who triggered it, target, linked
     job ID (clickable), current status badge.

What action or decision it supports:
  "What has the system been doing?" "Did my last action work?" "Are
  there any failed jobs I need to deal with?"

What data powers it:
  GET /api/ops/jobs/recent (returns last 50 from ops_jobs table)
  GET /api/ops/actions/recent (returns last 50 from ops_action_receipts table)
  Auto-refreshes every 15 seconds.

What states it uses:
  Job states: draft, queued, running, completed, completed_unverified,
  failed_retryable, failed_terminal, waiting_for_approval,
  waiting_for_dependency, fallback_mode, paused, stale, cancelled, skipped.
  Each state has a specific color badge.

What happens if data is stale:
  A "last refreshed X seconds ago" indicator shows at the top of each
  panel. If the API has not responded in 30+ seconds, the panel border
  turns amber with "Data may be stale" warning.

What happens if data is missing:
  If the ops_jobs table is empty (fresh install or after migration),
  the panel shows "No jobs recorded yet. Actions you take from this
  dashboard will appear here." — not a broken or blank state.

What happens if the underlying process fails:
  If a job fails, it appears in the table with a RED "FAILED" badge.
  The summary column shows the error reason. A retry button appears
  if the job is retryable.

What happens if the system is degraded or in fallback mode:
  A banner appears at the top of the manager dashboard: "System operating
  in degraded mode — [specific component] is running in fallback."
  Jobs created during degraded mode have fallback_indicator=true and
  show a yellow "FALLBACK" badge.

  "What is the state of the entire operation right now?" This is your

What data powers it:
  GET /api/ops/jobs/recent
  GET /api/ops/approvals/pending
  GET /api/ops/events/timeline?hours=24
  GET /api/ops/agents/status
  GET /api/ops/alerts/active
  GET /api/ops/actions/recent
  Plus existing /api/health for infrastructure panels.
  Auto-refreshes every 15 seconds.

What states it uses:
  Each panel uses its own canonical state model (see Section A).

What happens if data is stale:
  Each panel has its own freshness indicator. If any panel cannot reach
  its API, that panel shows an amber border with "Unable to reach [source]

What happens if data is missing:
  Each panel has a meaningful empty state. Approval Queue: "No pending
  approvals." Alerts: "No active alerts — system healthy." Jobs: "No
  recent jobs recorded."

What happens if the underlying process fails:
  If PostgreSQL is unreachable, the NOC shows a red infrastructure
  health warning (existing behavior) AND each ops panel shows "Database
  unreachable — operational data unavailable." If only one agent is
  down, the agent health panel shows that specific agent as UNREACHABLE
  while other panels continue working normally.

What happens if the system is degraded or in fallback mode:
  The NOC header shows a degradation banner. Affected agent cards show
  their degraded mode. Jobs running in fallback show the fallback badge.
  A "system degraded" alert appears in the alerts panel with details
  about which capability is affected and what quality impact exists.

G2: WHAT RUNS IN PARALLEL DURING MIGRATION
------------------------------------------------------------------------

Phase A — Shadow Mode (first 1-2 days of build):

  New ops_* tables exist in PostgreSQL but are NOT the primary source.
  Old systems (JSON jobs, in-memory events) continue as primary.
  truthful_ops.py writes to ops_* tables IN ADDITION TO old stores.
  This means every job is recorded in both places.

  Dashboard still reads from old sources during this phase.
  No visible change to the operator.

  Behind the scenes, a comparison log records:
  - Does ops_jobs match jobs.json for the same job_id?
  - Does ops_events match the in-memory event list?
  - Are there discrepancies?

Phase B — Parallel Display (next 1-2 days):

  New dashboard panels are added, reading from ops_* tables.
  Old panels remain visible alongside new panels.
  The operator can compare old view vs new view side by side.

  For example:
  - Old "event feed" section still shows in-memory events
  - New "event timeline" panel shows PostgreSQL events
  - They should show the same data — if they differ, we investigate

Phase C — Primary Cutover (after verification):

  Dashboard panels switch to reading exclusively from ops_* tables.
  Old JSON write continues as backup (dual-write maintained).
  Old panels removed or collapsed.

  The operator sees the new truthful panels as the primary surfaces.

Phase D — Cleanup (after owner sign-off):

  JSON file writes disabled (read-only archive).
  Old in-memory event list reduced to a short read-through cache.
  Redis data stores for jobs/events cleaned up.
  Only PostgreSQL remains as operational truth.

------------------------------------------------------------------------
G3: HOW OLD VS NEW TRUTH IS COMPARED
------------------------------------------------------------------------

During Phase A and B, a comparison script runs after every dispatch:

  1. Read the job from ops_jobs (new PostgreSQL store)
  2. Read the same job from jobs.json (old JSON store)
  3. Compare: job_id, status, agent, action, timestamps
  4. Log any discrepancy to a comparison_log.json file

  Expected discrepancies (acceptable):
  - ops_jobs has richer data (approval_id, correlation_id, risk_class)
    that the old store does not have — this is expected.
  - Timestamps may differ by milliseconds — acceptable.

  Unexpected discrepancies (investigate):
  - Job exists in one store but not the other
  - Status differs between stores
  - Agent or action differs

  I will report the comparison results at each staged review point.

  2. Zero unexpected discrepancies between old and new stores
  3. At least 20 jobs successfully recorded in ops_jobs
  4. At least 50 events successfully recorded in ops_events
  5. Dashboard panels reading from ops_* are rendering correctly
  6. Owner has reviewed the parallel display and confirmed new panels
